## 2026-04-22（P0：persona 横向链路打通 — 日文客群真正生效）

### 背景

4/21 完成的日本女性 37-60 岁客群（`jp_female_midlife`）改造**停留在"配置和前端"层**：

- `config/fb_target_personas.yaml` 有 persona + referral_priority
- `config/chat_messages.yaml.countries.jp` 有日文模板
- `config/facebook_playbook.yaml` 的 phase 覆盖有日本化默认值
- 前端命令台按 persona 显示 🇯🇵 徽章 + LINE 优先输入框
- router 层 `/referral-config` 和 `/active-persona` 都正确

但 `automation/facebook.py` 层的 **6 个核心业务方法**（add_friend / group_engage / extract_members / check_messenger_inbox / check_message_requests / check_friend_requests_inbox）**根本没读这些新配置** — 跑起来还是老行为，4/21 的工作等于"纸面合规"。

### 交付清单

#### 1. 新建 `src/app_automation/fb_content_assets.py`（单一文案包 helper，218 行）

**接口**（纯函数 + 异常兜底）：

- `get_verification_note(persona_key, name, interest_hint)`: 加好友验证语
- `get_greeting_message(persona_key, name)`: Messenger 开场白
- `get_comment_pool(persona_key, min_size=3)`: 群评论模板池
- `get_referral_snippet(channel, value, persona_key)`: LINE/IG/WA/TG 切换话术
- `describe_assets_for_persona(persona_key)`: 诊断接口（单测用）

**读取优先级**：
1. `chat_messages.yaml.countries[cc]`（YAML 真配置，热加载）
2. `_LOCALE_FALLBACKS[lang]`（内置日/意/英兜底，保证业务不崩）

**设计动机**：把"persona → country → 文案"映射收敛到一个 module，automation 方法只要多写 1 行 helper 调用，避免 6 处重复读 yaml + 映射逻辑。

#### 2. `src/app_automation/facebook.py` 接入 5 条 automation 业务线

新增内部 helper `_resolve_phase_and_cfg(section, device_id, phase_override)` 把 "`get_phase` + `resolve_params`" 两步合一，业务方法调一行即可。

| `extract_group_members` | `resolve_extract_members_params(phase)` | `target_country` 空时从 persona 派生 | `max_members=0` → 直接 return [] |
| `check_messenger_inbox` | `resolve_check_inbox_params(phase)` | `persona_key` 记入 stats | `max_conversations=5` |

### 关键行为变化

1. **冷启动新号绝对不加好友**：`add_friend_with_note` 在 `cold_start` 阶段直接 `return False`，不再走 UI 路径。
2. **日本女性加好友必带日文验证语**：即使前端/任务参数完全不填 note，也会从 `friend_request_notes` 随机抽一条日文填入。
3. **群内评论不再发英文**：`comment_pool` 未传时从 `countries.jp.comment_templates`（4 条日文）随机。
4. **引流切换话术 LINE 优先**：`get_referral_snippet('line', value)` 返回日文拼好的完整句子（含 emoji 🌿）。
5. **冷却期 extract_members 完全禁**：`max_members=0` → 直接返回空列表，不进群、不 dump 列表。

### 深度二次优化结果

原计划是"每个业务方法 inline 读 yaml"，实施中改为"单独建 `fb_content_assets.py` helper"。好处：

- automation 层改动最小（每方法 ~5 行）
- 未来扩展新国家客群 = 只改 YAML + 新增 persona 节点，automation 零改动
- 纯函数易单测，不依赖 device_manager / database

缺陷纠正：第一次 smoke 暴露 **playbook yaml 字段名** (`max_posts` / `max_members`) 和 **automation 代码字段名** (`max_posts_per_group` / `max_members_per_run`) 不一致 — 就地对齐为 yaml 版本。

- `FacebookAutomation._vlm_ocr_profile_texts(image_paths)`：

### 四、本轮"再优化"三个判断

1. **不做"自动改 yaml"**：最初设计过「L1 precision 差的 rule 自动把 weight 调低」，但这会形成反馈环——
   一条 rule 一旦权重降了就更难再命中，更难拿到新样本，自我验证偏差。**改为只产出建议，留给人类决策**。
2. **D4 拆成两级阶梯而非一刀切**：硬暂停太激进（风控刚过 12h 立即全速）；软降档太宽松（严重风控也只减半）。
   **二级阶梯**在"保守"和"持续运营"之间取平衡。
3. **兴趣去重的语义**：纠结过是否把 `ヨガ` 和 `yoga` 合并成同一 topic。最终**不合并**——
   片假名 vs 罗马字是不同的语言层表达，后续"相似帖推荐"时日文帖和英文帖的检索策略本就不同，
   合并反而丢失信号。只做"yoga/YOGA/Yoga"这类拉丁大小写合并。

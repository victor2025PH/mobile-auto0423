# 集群节点与任务回收站说明

> 给人看的运维备忘；与代码行为一致时的摘要。

## 一、回收站数据存在哪里

- 软删除字段为 SQLite 表 **`tasks.deleted_at`**（由 `database` 迁移添加）。
- **`GET /tasks?trash_only=true`**、**`GET /tasks/count?trash_only=true`** 仅查询**当前节点**本地库，**不会**与集群 Worker 的 `/tasks` 结果合并（与主列表不同）。
- **`GET /tasks/{id}?include_deleted=true`**：本地无记录时，会依次请求各 Worker 的同名接口；若 Worker 也带 **`include_deleted`**，主控会转发该 query（便于节点版本一致后排查）。

## 二、多节点部署时要注意什么

1. **每个节点**都要跑同一套迁移，保证存在 **`deleted_at`** 列与索引，否则该节点任务相关 API 会报错。
2. **代码版本**：主控与 Worker 的 `task_store` / `tasks` 路由应大致对齐，避免一端仍物理删、一端软删导致行为不一致。
3. **回收站不是跨节点聚合**：在 A 节点软删的任务，在 B 节点的「回收站」里**看不到**，除非未来单独做同步或集中存储。

## 三、建议验证

- 升级后在本机执行：`GET /tasks/count?trash_only=true`、软删一条再查计数是否 +1。
- 自动化：`tests/e2e/test_trash_recycle_bin.py`（需 Playwright）。**注意**：用例先请求 `/health` 再调 `POST /tasks/erase-batch`，**再在数据校验后**打开 `/dashboard` 做轻量冒烟；若先加载完整仪表盘（大量脚本与连接），在 **单 worker uvicorn** 下可能与同进程的 `erase-batch` 并发导致接口长时间挂起，属环境限制而非业务逻辑错误。

## 四、可选后续

- 主控/开发机跑 E2E 时可将 uvicorn **workers>1** 或单独排查仪表盘首屏是否有同步阻塞，以便恢复「先开任务页再调 API」类用例。
